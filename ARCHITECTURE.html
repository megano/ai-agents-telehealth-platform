The Care Coordination Agent care_coordination_agent.py

Event-driven, proactive care team planning for maternity and postpartum journeys. The Transition Coordinator (TC) wakes on stage milestone events and orchestrates a pipeline that selects, validates, and introduces a patient's care team, before problems arise, not after.

Demo scenario: Sofia Nguyen (TX · 26 weeks · T2) triggers a T2→T3 stage transition. The TC plans her third-trimester care team, validates state licensure and patient preferences, then drafts a warm patient-facing care team introduction.

Provider Filter: Logic, Not LLM

The provider filter is a pure Python function, no model call. It enforces hard constraints that don't require reasoning:

FilterRule
State licenseProvider's licensed_states must include patient's state (or be "all" for non-licensed roles like coaching)
Availabilityaccepting_new_patients = true
Specialty matchProvider specialty maps to at least one provider_type required by the target stage interventions

Validator: 6 Hard Checks

#CheckWhat it catches
1state_license_validRecommended provider not licensed in patient's state
2accepting_new_patientsProvider has closed panel, filter missed it or candidate was updated
3no_hallucinated_idsPlanner invented a provider_id not in the candidate list
4continuity_preservedProvider with continuity_required=true from current stage dropped without explanation
5unmet_needs_flaggedRequired provider type has no candidate but unmet_needs is empty, silent gap
6language_preference_metPatient requires a specific language but no provider in the plan speaks it

Care Pathway Design: Proactive vs Reactive

The standard maternity pathway spans four stages (T1 → T2 → T3 → Postpartum). Each stage defines required provider types, interventions, evidence_level (established or emerging), and a proactive_trigger_week : the week the TC initiates planning, before the stage begins.

StageWeeksProactive provider introductions
T11-12OB-GYN · Registered Dietitian (GD prevention) · Therapist · Career Coach (leave planning)
T213-27OB-GYN · Pelvic Floor PT · Dietitian (GD screening) · Career Coach (handoff planning)
T328-40OB-GYN · Pelvic Floor PT · Lactation Consultant (LC) intro · Pediatrician intro · Career Coach (RTW plan)
Postpartum0-12 ppOB-GYN · LC · Pelvic Floor PT · Therapist · Career Coach (re-entry support)
Design principle, proactive over reactive: The LC and pediatrician are introduced in T3 (weeks 34-36), before birth. Evidence shows meeting the LC prenatally significantly improves breastfeeding duration. The career coach enters at T1 to design the leave plan, not at postpartum when re-entry anxiety is already acute.

Data Access Boundaries

Design principle: Agents receive only the minimum data needed for their task. Sensitive identifiers are stripped before any LLM prompt is constructed. Clinical data is never in scope.
Data Care Coordination Agent Notes
Patient first name ✅ Patient intro greeting only
Care stage + pregnancy week ✅ Drives TC stage transition logic
Patient state (location) ✅ Hard constraint for state licensure check
Matching preferences (gender, language, modality) ✅ Passed to Planner for preference-aware selection
Provider name + credentials ✅
Provider licensed_states ✅ Filter and validator use this, never sent to LLM raw
Care pathway templates ✅ Stage definitions, intervention types, evidence levels
Patient date of birth ❌ Not exposed to any agent
Insurance ID / plan ❌ Planned Insurance Agent handles via secure API, never in LLM prompt
Clinical notes / diagnoses ❌ HIPAA boundary, permanently out of scope
Billing / payment data ❌ Out of scope for all agents in this demo

Planned Agents: Future Roadmap

Designed in the architecture but not yet implemented. Each is called out to show intentional product thinking about what comes next.

FUTURE

🏥 Insurance Eligibility Agent

Trigger: Before care plan is confirmed

Purpose: Verify patient insurance covers each recommended provider and visit type before the care plan is sent to the patient

Design note: Insurance data handled via secure API (e.g. Availity), never passed to LLM prompt. Agent receives only eligibility verdict + estimated cost.

FUTURE

📧 Post-Visit Follow-Up Agent

Trigger: 2 hours after visit completion

Purpose: Generate post-visit summary with next steps and stage-appropriate resources

Validator checks: No clinical diagnosis language, correct provider sign-off, resource links valid, language matches patient preference

FUTURE

📱 SMS Reminder Agent

Trigger: T-72h, T-24h, T-2h before appointment

Purpose: Send structured reminders; handle CONFIRM / CANCEL / RESCHEDULE replies

Guardrail: Hard block if patient on opt-out list, no retry, no override

FUTURE

🔀 Pathway Variant Engine

Trigger: Clinician flags risk condition at intake or screening

Purpose: Extend standard maternity pathway with variant-specific providers (GD, mental health risk, high-risk OB)

Design note: Variant schemas already modeled in care-pathways.json. Activation is always clinician_flagged_need, never auto-triggered by agent.

Validator Pattern

The validator is deterministic Python, not a model. The six checks are factual lookups (state-license tables, set membership, boolean fields), so they run in code and are guaranteed correct and reproducible. Results are typed Pydantic (ValidationResult) for a clean audit trail and unambiguous routing.

# Typed result the deterministic checks return class ValidationResult(BaseModel): passed: bool checks: list[ValidationCheck] # named check + pass/fail + detail failures: list[str] # empty if passed=True fix_instructions: str # fed back to upstream agent on retry # Router uses typed result, no parsing, no ambiguity def route_after_validation(state) -> str: if state.validation.passed: return "draft" if state.validation_attempts >= MAX_RETRIES: return "block" return "replan"
Why this matters: In agentic systems that touch patient care decisions, "probably correct" isn't acceptable. Structured validation with a hard block path ensures the system fails safely, routing to a human staff queue rather than sending a flawed care plan to a patient.
